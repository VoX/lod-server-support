# LSS Simplification & Maintainability Review

**Date:** 2026-06-09 · **Branch:** `paper-release-26.1.2` (working tree incl. uncommitted round-3 hygiene changes — all `file:line` anchors refer to that state) · **Status: decision document — nothing here is implemented.**

A multi-agent critical design review of how LSS works and is implemented, hunting three things the maintainer asked for: places to **simplify** the implementation, **unneeded code or functionality to remove**, and **refactors that improve readability/maintainability**. Method: 3 mapping agents traced the server flow, client flow, and full class/config/payload inventory; 8 design critics with distinct lenses produced 42 raw findings; overlapping proposals were merged into 28; each was then independently, adversarially verified against the code (premise re-checked, hidden users grepped, threading/wire/mapping constraints attacked, value judged). **All 28 survived** — verifier corrections are preserved per finding under *Caveats & corrections*.

## How to read this

- **Worth** is the verifier's honest "would a pragmatic maintainer do this?" — `high` = clear win (13 findings), `medium` = real but discretionary (15). Nothing rated `low` survived.
- **Flags**: `protocol-breaking` = changes wire bytes, requires a PROTOCOL_VERSION bump (bundle these); `behavior-change` = observable behavior shifts (described in the finding); `user-facing-removal` / `config-change` = admins/users notice.
- Findings are independent unless the **Sequencing & interactions** section says otherwise.
- Each finding keeps the verifier's *Caveats & corrections* — read them; several correct overstated claims in the original critique and they calibrate trust in the rest.

## Sequencing & interactions

**Bundle A — one protocol v16 bump.** The wire-breaking findings should land together in a single `PROTOCOL_VERSION` bump rather than dribble across releases: #2 `removable:F2` (delete cancellation) **or** #19 `protocol:F4` (batch cancels) — these are *alternatives, pick one*; #13 `protocol:F1` (key the protocol by position — the largest change, and it subsumes cancel addressing, so decide it first); #20 `protocol:F3` (drop SessionConfig write-only fields); #21 `protocol:F5` (drop dual dimension encoding); the phase-2 wire part of #3 `yagni:F3`. Note #1 `protocol:F2` tier 1 (delete the orphaned bandwidth receive pipeline) is wire-invisible and can ship any time.

**Bundle B — concurrency/ownership cluster.** #6 `concurrency:F2`'s step (a) (transfer snapshot map ownership — trivial) unblocks #7 `concurrency:F3` (generation outcomes via the snapshot) and #15 `concurrency:F4` (fold the side queues). #5 `concurrency:F1` (derive permits from the pending maps) and #8 `concurrency:F5` (dimension change = remove+re-register) are independently applicable; #5 gets simpler if #14 `removable:F4` (delete the waitingQueue) is accepted first — and #14 also removes the only server-side use of the rate-limit knobs, simplifying #3 `yagni:F3` phase 1. Decide #14, then #5, then the rest.

**Bundle C — common/ hoists (cheap, parity-test-protected).** #9 `server-design:F3` (the investigation doc's sanctioned Phase-1 send-path hoist), #10 `duplication:F3` (config defaults/clamps), #11 `duplication:F1` (ReadResult record), #18 `duplication:F5` (command bodies). Independent of everything else; all stay MC-free.

**Bundle D — client.** #22 `client-design:F1` (consolidate per-column state) first; #23–#26 touch the same classes and get easier after it. Fully independent of the server bundles.

**Standalone quick wins** (small, low-risk, no wire change): #4 `api-config:F4` (gate consumer-less clients at the source), #16 `server-design:F4` (stop re-passing fixed router deps), #17 `server-design:F6` (delete the 1-implementer interface), #12 `server-design:F5` (unify the duplicated result ladder), #27 `yagni:F5` (keep the benchmark out of the prod jar), #28 `api-config:F5` (remove the off-thread toggle).

## Part 1 — High-value findings

### 1. Remove the orphaned bandwidth-negotiation pipeline: the client sender was deleted before any release, but both servers still implement the full receive path (BandwidthUpdateC2S + desiredBandwidth + SessionConfig.playerBandwidthLimit)

**ID** `protocol:F2` (protocol & wire; merged: `api-config:F1`, `client-design:F2`, `removable:F1`) · **Category** remove · **Worth** high · **Confidence** 0.92 · **Flags** protocol-breaking

**Files:** `fabric/src/main/java/dev/vox/lss/networking/payloads/BandwidthUpdateC2SPayload.java`, `fabric/src/main/java/dev/vox/lss/networking/LSSNetworking.java`, `fabric/src/main/java/dev/vox/lss/networking/server/LSSServerNetworking.java`, `fabric/src/main/java/dev/vox/lss/networking/server/RequestProcessingService.java`, `common/src/main/java/dev/vox/lss/common/processing/AbstractPlayerRequestState.java`, `common/src/main/java/dev/vox/lss/common/LSSConstants.java`, `fabric/src/main/java/dev/vox/lss/networking/payloads/SessionConfigS2CPayload.java`, `paper/src/main/java/dev/vox/lss/paper/LSSPaperPlugin.java`, `paper/src/main/java/dev/vox/lss/paper/PaperPayloadHandler.java`, `paper/src/main/java/dev/vox/lss/paper/PaperRequestProcessingService.java`

**Current state & critique.** One of the protocol's 8 messages, BandwidthUpdateC2SPayload, retains a complete receive pipeline on both platforms — Fabric registration (LSSNetworking.java:29-32) + receiver (LSSServerNetworking.java:97-105) + handler (RequestProcessingService.java:134-138); Paper channel (LSSPaperPlugin.java:35), dispatch (:97), handler (:162-168), decoder (PaperPayloadHandler.java:170-175) — feeding AbstractPlayerRequestState.desiredBandwidth (:57, :290-291), consumed in both flushSendQueues (RequestProcessingService.java:263, PaperRequestProcessingService.java:273). But the only client sender ever written (LodRequestManager, 0.1.0-dev) was deleted in commit 6ddc9eb together with the ConcurrencyController that computed the rate; no tagged release (v0.2.1 through current) contains any send site, so desiredBandwidth has been Long.MAX_VALUE in every shipped build and `Math.min(perPlayerCap, Math.max(1, getDesiredBandwidth()))` has always reduced to plain perPlayerCap. Its S2C counterpart, SessionConfigS2CPayload.playerBandwidthLimit (field 10 of 10, populated at LSSServerNetworking.java:57 / LSSPaperPlugin.java:126), is likewise never read by the client (zero accessor calls outside two tests). Every reader of the bandwidth path must reason about a negotiation that cannot occur, and both wire-parity suites must keep pinning a message nobody sends; server-side throttling is already fully covered by SharedBandwidthLimiter's global/per-player caps plus client-side queue backpressure.

**Proposal.** Tier 1 (wire-invisible, no protocol bump strictly required): delete BandwidthUpdateC2SPayload.java (25 lines); the Fabric registration (LSSNetworking.java:29-32 + import :3) and receiver (LSSServerNetworking.java:97-105 + import :6); RequestProcessingService.handleBandwidthUpdate (:134-138 + import :17); the Paper channel registration (LSSPaperPlugin.java:35), dispatch case (:97), and handleBandwidthUpdate (:162-168); PaperPayloadHandler.decodeBandwidthUpdate + DecodedBandwidthUpdate (:170-175); PaperRequestProcessingService.handleBandwidthUpdate (:138-143); LSSConstants.CHANNEL_BANDWIDTH_UPDATE (:21); the desiredBandwidth field + accessors (AbstractPlayerRequestState.java:57, :290-291); and the Math.min/max term in both flushSendQueues (RequestProcessingService.java:263, PaperRequestProcessingService.java:273 — both become plain perPlayerCap). This is safe against any stray pre-release client: the v15 handshake gate already excludes them, the handler no-ops pre-handshake, and Fabric/Bukkit silently drop payloads on unregistered channels. Tier 2 (requires protocol bump 15→16, do in the same change since client and server ship in lockstep): drop playerBandwidthLimit from SessionConfigS2CPayload (record 10→9 fields, codec write/read lines :36/:48), both construction sites (LSSServerNetworking.java:57, LSSPaperPlugin.java:126), and the PaperPayloadHandler encodeSessionConfig/sendSessionConfig parameters (:40/:51, :60/:65). Update 4 test files (delete fabric WireParityTest:107-114, PayloadCodecTest:106-115, paper WireParityTest:93-101; trim the SessionConfig fixtures in all four) and the protocol list at CLAUDE.md:65. The server config key bytesPerSecondLimitPerPlayer is untouched — it still drives perPlayerCap (RequestProcessingService.java:259). Optional rider for the maintainer: serverCapabilities and generationRateLimitPerPlayer are also client-unread today, but serverCapabilities is the protocol's forward-compat negotiation mechanism (its C2S twin IS read at registerPlayer) — recommend keeping both and treating further field pruning as a separate decision. If client bandwidth preference is ever wanted again, the client-side rate-derivation logic would have to be rebuilt from scratch regardless (it died with ConcurrencyController in 6ddc9eb), so re-adding the message then is an ordinary protocol bump.

**Impact.** ~85 main-source lines deleted across 10 files (whole payload class, one of four Fabric serverbound registrations, one Fabric receiver, one Bukkit channel, two handlers, one decoder+record, one channel constant, one volatile field + accessors, SessionConfig 10→9 fields) plus ~30 test lines across 4 test files and 1 doc line; protocol shrinks 8→7 messages; the per-player send-budget computation becomes a single obvious min(globalAllocation, configCap) with no phantom negotiation input; one fewer network-thread→tick-thread volatile handoff to reason about in the bandwidth path.

**Risk.** Low. Pure deletion; no shipped client ever emits the message (verified across all release tags), the v15 handshake gate plus silent unknown-channel dropping on both platforms covers even hypothetical pre-release jars, and the SessionConfig field drop ships in lockstep behind the protocolVersion check. Parity coverage for the remaining 7 payloads is untouched — the bandwidth test methods are deleted, not weakened.  
**Effort.** Small — mechanical deletions across 14 files + CLAUDE.md line + protocol version bump; no new code, no behavior change to write tests for.

**Caveats & corrections (verifier).** (1) Premise correction: "zero send code in fabric client history" is false — a client sender existed in 0.1.0-dev (tag before-loop, branches chunk-ingest/research; LodRequestManager sent the payload with 10%-change hysteresis) and was deliberately removed in commit 6ddc9eb alongside ConcurrencyController. No tagged/published release ever sent it; the correct framing is "orphaned receive pipeline", which strengthens the case (the client-side machinery to compute a desired rate is already gone, so the receiver can never be exercised without rebuilding that feature anyway). (2) Touches 4 test files, not "2 test suites": fabric WireParityTest:107-114 + PayloadCodecTest:106-115 (+fixtures :54/:67), VoxelSectionPayloadTest (:130/:140 fixtures), paper WireParityTest:93-101 (+fixture :106-121). (3) The original's config-change flag is wrong — no user config key changes; bytesPerSecondLimitPerPlayer stays and still drives perPlayerCap. (4) Only the SessionConfig field drop is protocol-breaking; the C2S pipeline removal is wire-invisible and safe standalone (no shipped client emits it; Fabric/vanilla discards unknown custom payloads, Bukkit drops unregistered channels, and the handler already no-ops pre-handshake). (5) Worst case for stray 0.1.0-dev jars: their messages are silently dropped and those clients are already locked out by the v15 handshake gate.

### 2. Request cancellation is dead protocol weight: it cannot cancel anything, distorts the concurrency limiters, and exists only to justify the pendingByRequestId twin map

**ID** `removable:F2` (removable functionality) · **Category** remove · **Worth** high · **Confidence** 0.9 · **Flags** protocol-breaking, behavior-change

**Files:** `fabric/src/main/java/dev/vox/lss/networking/payloads/CancelRequestC2SPayload.java`, `fabric/src/main/java/dev/vox/lss/networking/client/LodRequestManager.java`, `fabric/src/main/java/dev/vox/lss/networking/client/InFlightTracker.java`, `fabric/src/main/java/dev/vox/lss/networking/server/LSSServerNetworking.java`, `fabric/src/main/java/dev/vox/lss/networking/server/RequestProcessingService.java`, `common/src/main/java/dev/vox/lss/common/processing/AbstractPlayerRequestState.java`, `common/src/main/java/dev/vox/lss/common/processing/OffThreadProcessor.java`, `common/src/main/java/dev/vox/lss/common/processing/PlayerStateAccess.java`, `common/src/main/java/dev/vox/lss/common/processing/IncomingRequestRouter.java`, `paper/src/main/java/dev/vox/lss/paper/LSSPaperPlugin.java`, `paper/src/main/java/dev/vox/lss/paper/PaperPayloadHandler.java`, `paper/src/main/java/dev/vox/lss/paper/PaperRequestProcessingService.java`

**Current state & critique.** The client sends CancelRequestC2SPayload on movement-prune beyond lodDistance+32 and on dimension change / cache flush (LodRequestManager.java:366-377, InFlightTracker.java:129-141), but the server's entire response is OffThreadProcessor.java:287-294: remove the pending entry and release a permit. Tracing every path shows it cancels nothing: requests still in the waiting queue or incoming queue have no pending entry (pendings are created only at submit — IncomingRequestRouter.java:244/:259, OffThreadProcessor.java:397) and the cancel drain runs before routing each cycle, so they are submitted and served anyway; already-submitted disk reads and generations are never aborted — results drain and the column is still built and sent via the requestId fallback at OffThreadProcessor.java:325/:419, and queued generation tickets are submitted without any pending check (RequestProcessingService.java:326-345). The only genuine effect is narrow and accidental: a cancel landing mid-disk-read suppresses the disk-miss→generation fallback (handleDiskNotFound requires pending != null at :393). Meanwhile the path actively distorts accounting: the permit released at :292 is released again when the result drains (:327/:421; ConcurrencyLimiter.release floors at 0, so the cap transiently rises by one), and :292's forRequest(GENERATION, diskReadingAvailable=true) maps a gen-fallback pending — which holds a generation permit — to the syncOnLoad limiter (RateLimiterSet.java:36-40), the only site in the codebase that can release a limiter mismatched with the permit held. Carrying this near-no-op costs a wire channel, a per-player cross-thread queue, and the pendingByRequestId twin map with four hand-synced sites (AbstractPlayerRequestState.java:50,198-202,209-212,218-226,172) — exactly the class of manually-synchronized permit state where this codebase's prior leak bugs lived — while the common staleness case (the 10s timeout sweep, InFlightTracker.java:111-123) deliberately doesn't cancel at all.

**Proposal.** Delete the cancellation path end-to-end, bundled with the next PROTOCOL_VERSION bump (currently 15, LSSConstants.java:11). Fabric: CancelRequestC2SPayload.java (24 lines), registration LSSNetworking.java:25-28, receiver LSSServerNetworking.java:87-95, RequestProcessingService.handleCancel:128-132; client: sendCancelPacket/pruneAndCancelOutOfRangePending/cancelAllPending (LodRequestManager.java:366-377, with onDimensionChange:345 and flushCache:393 dropping their calls and :146 calling tracker.pruneOutOfRange directly), InFlightTracker.forEachRequestId:146-150 and the IntConsumer parameter of pruneOutOfRange:129. Paper: LSSPaperPlugin.java:34 channel registration, :96 switch case, handleCancelRequest:153-160; PaperPayloadHandler.java:163-167; PaperRequestProcessingService.handleCancel:131-136. Common: the OffThreadProcessor.java:287-294 drain block; incomingCancels queue + counter + addCancel:103-109 + pollCancel:243-248 + the clears at :159/:161; the pendingByRequestId map (:50) and its sync lines in addPendingRequest/removePendingByPosition/clearProcessingState (:198-202,:209-212,:172); removePendingByRequestId:218-226; PlayerStateAccess.java:29,:33. The one other removePendingByRequestId caller, IncomingRequestRouter.java:251, switches to removePendingByPosition(req.cx(), req.cz()) — exactly equivalent, since the pending it unwinds was just added keyed by that position at :244. Remove CHANNEL_CANCEL_REQUEST (LSSConstants.java:20), the three test sections (fabric WireParityTest:99-106, paper WireParityTest:82-90, PayloadCodecTest:94-103), and the two CLAUDE.md mentions (:65,:119). Stale-pending garbage collection already exists without cancel: results always drain (with the requestId fallback), dimension change clears pending state and resets limiters (AbstractPlayerRequestState.java:170-183), and disconnect drops the player state. Accepted behavior change: the narrow disk-miss→generation-fallback suppression is lost, so a few extra fallback generations run for chunks pruned during fast travel — bounded by the generation concurrency cap and rate limiter, and their output still populates the timestamp cache and region files. If early-abort is ever genuinely wanted, it must be built as a real feature (position-keyed, able to pull waiting-queue entries and abort pool tasks/generation tickets before submission) under a protocol bump — the current packet structurally cannot do any of that.

**Impact.** ~135 main-source lines across 14 files (10 fabric/common + 4 paper) plus ~25 test lines and 2 doc lines removed; one wire channel, one per-player cross-thread queue (incomingCancels CLQ + counter), and one whole protocol concept eliminated. Pending tracking collapses from a hand-synced twin map (pendingByRequestId + pendingByPosition, 4 sync sites) to a single position-keyed map, structurally removing one invariant of exactly the kind that produced this project's prior permit-leak bugs. Deletes the only out-of-band permit release in the codebase — fixing by removal the cancel double-release (transient cap+1 under-count) and the only site that can release a limiter mismatched with the permit held (gen-fallback pending → syncOnLoad). Also stops the client sending one unbatched cancel packet per pruned request during fast travel.

**Risk.** Low. Server behavior is unchanged except (a) permits are now released only when results drain — strictly more correct accounting — and (b) the narrow loss of disk-miss→generation-fallback suppression, bounded by the generation concurrency/rate limiters and partially recouped via timestamp-cache/region-file population. Cross-version tolerance is good even without a bump (Fabric ignores unregistered serverbound custom payloads as DiscardedPayload; Paper's Messenger drops unregistered incoming channels), and the existing handshake version gate already refuses mismatched sessions; bundling with the next PROTOCOL_VERSION bump removes all doubt. No constraint violations: pure deletion keeps common/ MC-free, removes (not shares) fabric/paper twin code — shrinking the drift surface the shared-base investigation worries about — and WireParityTest sections are deleted symmetrically on both platforms.  
**Effort.** Medium: mechanical deletions across 17 files (14 main + 3 test) on two platforms plus a one-line router swap and two CLAUDE.md edits; should ride along the next planned protocol bump rather than force one.

**Caveats & corrections (verifier).** The 'functional no-op' premise is overstated in one sub-path: a cancel drained while a disk read is in flight removes the pending, so a subsequent notFound result for a GENERATION-type request skips the generation fallback (OffThreadProcessor.java:393 requires pending != null). Removing the path means those fallback generations run — bounded by the generation concurrency cap and rate limiter, and their output populates the timestamp cache and region files, but it is a real behavior change (behavior-change flag is correct, though it covers more than just 'beneficial' accounting changes). The dimension-change claim is directionally right but imprecise on ordering: cancels typically arrive after the server-side wipe (server detects the dimension change before the client reacts) and then drain against already-cleared pending maps — same no-op outcome, different mechanism. Line counts are mildly inflated: ~135 main-source lines (not ~150) across 14 main files (not 12) plus ~25 test lines and 2 CLAUDE.md doc lines. Channel removal without a PROTOCOL_VERSION bump is tolerated in practice (Fabric: unregistered serverbound custom payloads become DiscardedPayload and are ignored; Paper: the Messenger drops unregistered incoming channels), but bundling with the next bump is the right call and the handshake version gate already rejects mismatched sessions.

### 3. The two 'RateLimitPerPlayer' knobs are documented as requests-per-second but no server-side rate enforcement exists — they only size a backlog queue (plus a client budget tie-break inert at defaults), and SessionConfig ships a behaviorally dead serverCapabilities field

**ID** `yagni:F3` (speculative generality; merged: `api-config:F2`) · **Category** remove · **Worth** high · **Confidence** 0.92 · **Flags** protocol-breaking, config-change, user-facing-removal, behavior-change

**Files:** `fabric/src/main/java/dev/vox/lss/config/LSSServerConfig.java`, `paper/src/main/java/dev/vox/lss/paper/PaperConfig.java`, `common/src/main/java/dev/vox/lss/common/processing/RateLimiterSet.java`, `common/src/main/java/dev/vox/lss/common/processing/IncomingRequestRouter.java`, `fabric/src/main/java/dev/vox/lss/networking/payloads/SessionConfigS2CPayload.java`, `paper/src/main/java/dev/vox/lss/paper/PaperPayloadHandler.java`, `fabric/src/main/java/dev/vox/lss/networking/client/SpiralScanner.java`, `common/src/main/java/dev/vox/lss/common/LSSConstants.java`, `README.md`

**Current state & critique.** README.md:106/108 documents syncOnLoadRateLimitPerPlayer=800 and generationRateLimitPerPlayer=80 as "requests per second per player", but the server enforces no per-second request limit: the only server-side read of either value is IncomingRequestRouter.java:182, where their sum (880) caps the waiting-queue backlog — a quantity with no time dimension, drained by permit release, while the codebase's only time-based limiting is byte token buckets (PlayerBandwidthTracker, SharedBandwidthLimiter; confirmed via git -S back to the knob's introduction in 0beb4e4, which never had per-second semantics). Client-side, generationRateLimitPerPlayer is decoded and never read (zero main-code accessor calls), and syncOnLoadRateLimitPerPlayer only clamps the once-per-second scan budget at SpiralScanner.java:57-60 via min(syncConcurrency*4, syncRate) — a tie (800 vs 800) at shipped defaults, binding only in non-default configs, and unenforced server-side either way (plus one debug log at LSSClientNetworking.java:89). Separately, SessionConfigS2CPayload.serverCapabilities (SessionConfigS2CPayload.java:13/:30, PaperPayloadHandler.java:45) is always the constant CAPABILITY_VOXEL_COLUMNS and has zero behavioral readers — only the C2S capability direction is load-bearing (AbstractPlayerRequestState.java:80-82) — and the exact-version handshake gate (LSSServerNetworking.java:60) means capability evolution coincides with protocol bumps anyway, mooting the field's forward-compat rationale. The fake knobs cost real surface: 2 constructor params threaded through registerPlayer on both platforms (RequestProcessingService.java:91-92, PaperRequestProcessingService.java:93-94) into AbstractPlayerRequestState.java:64-68 and RateLimiterSet (2 fields + 2 getters), MIN_/MAX_RATE_LIMIT (LSSConstants.java:60-61), clamp lines duplicated in both validate() methods, 2 clamp unit tests, and 2 gametest assertions.

**Proposal.** Phase 1 (no wire change): remove both options from LSSServerConfig.java:20,22,42,44 and PaperConfig.java:30,32,66,68; delete MIN_/MAX_RATE_LIMIT (LSSConstants.java:60-61); shrink RateLimiterSet to its two ConcurrencyLimiters (49→~33 lines, dropping 2 fields/2 getters/2 ctor params); drop the syncRate/genRate params from AbstractPlayerRequestState.java:64-68 and both registerPlayer call sites. For the waiting-queue cap at IncomingRequestRouter.java:182, pick one: (a) derive from concurrency — ConcurrencyLimiter.getMaxConcurrency() already exists (ConcurrencyLimiter.java:45-47), so 4*(syncOnLoad().getMaxConcurrency()+generation().getMaxConcurrency()) = 864 vs today's default 880 with zero new plumbing; or (b) one honestly-named requestQueueLimitPerPlayer knob (default 880) if admin backlog tunability matters — (b) is zero-behavior-change at defaults. **Files:** `fabric/src/main/java/dev/vox/lss/config/LSSClientConfig.java`, `fabric/src/main/java/dev/vox/lss/networking/client/ClientColumnProcessor.java`, `fabric/src/main/java/dev/vox/lss/config/LSSConfigMenu.java`, `fabric/src/main/java/dev/vox/lss/api/VoxelColumnConsumer.java`, `README.md`

**Current state & critique.** LSSClientConfig.offThreadSectionProcessing (LSSClientConfig.java:12, default true) is read at exactly one branch, ClientColumnProcessor.scheduleProcessing:80-96, invoked from END_CLIENT_TICK (LSSClientNetworking.java:212-219): true CAS-gates a drain onto the LSS-ColumnProcessor daemon; false runs drainColumnQueue (:99-153) inline on the main client thread, synchronously decoding and dispatching up to MAX_QUEUED_COLUMNS=8000 (:25) backlogged columns in a single tick — a freeze hazard after exactly the kind of burst the queue exists to absorb. Because the mode is user-switchable from the Sodium UI (LSSConfigMenu.java:58-68, README.md:126), VoxelColumnConsumer's threading contract is conditional (VoxelColumnConsumer.java:13-17) yet still ends with an unconditional 'Implementations must be thread-safe' — so no consumer can rely on main-thread dispatch, and the off mode guarantees nothing; worse, the inline branch bypasses the processing CAS, so flipping the live binding mid-drain can dispatch from two threads concurrently. All three registered consumers are off-thread-ready (VoxyCompat.java:48-61 feeds Voxy's queue-based rawIngest; LSSClientGameTests.java:15 and BenchmarkHook.java:54 are no-op lambdas), and nothing in tests, benchmark, or resources ever sets the flag false. git log -S traces the flag to the earliest bring-up commits ('wip', 'working?') — a development escape hatch that shipped as an end-user option.

**Proposal.** Delete the toggle: remove the field (LSSClientConfig.java:12), the Sodium option group (LSSConfigMenu.java:58-68) plus its two lang keys (assets/lss/lang/en_us.json:8-9), and the else-branch in scheduleProcessing (ClientColumnProcessor.java:80,94-96), keeping the CAS+executor path unconditionally; drop the README.md:126 row and the CLAUDE.md:135 mention. Tighten VoxelColumnConsumer's javadoc to one unconditional contract: 'onVoxelColumnReceived is always called on the LSS-ColumnProcessor background thread; implementations must be thread-safe.' If a maintainer debugging escape hatch is still wanted, gate the inline path on Boolean.getBoolean("lss.inlineColumnProcessing") — one line, zero user-facing surface. No migration needed: GSON ignores the stale JSON key on load and JsonConfig.load()'s immediate re-save drops it (verified in JsonConfig.java); add a changelog note for users who had pinned false. Client-only change: no wire, server, common/, or Paper impact, and no test updates required (no test references the flag).

**Impact.** Client config surface 3 -> 2 options; ~20 LOC of code removed (1 config field, 11-line Sodium UI group, 4-line branch, 2 lang keys) plus README/CLAUDE.md rows and a simpler javadoc; the public VoxelColumnConsumer API gets ONE threading model instead of a config-dependent pair (halving the dispatch matrix future consumers must reason about); eliminates a settings-screen-reachable worst-case render-thread stall (8000-column inline drain) and the toggle-mid-drain dual-thread dispatch wrinkle.

**Risk.** Low. Default true is the surviving path, so virtually all users see no change; stale config keys are verified harmless (GSON drops unknown fields, immediate re-save removes them). The only loss is user-facing: anyone who explicitly set false (e.g. while chasing a consumer bug) silently reverts to off-thread processing — mitigated by the optional -D escape hatch and a changelog note. No wire-format, protocol, threading-model, or module-boundary implications; aligns the client with its documented 'main thread + single processing executor' model.  
**Effort.** Small: single-module (fabric client) deletion across 5 code/resource files plus 3 doc touchpoints; no tests reference the flag, so none need updating; optional 1-line system-property hatch.

**Caveats & corrections (verifier).** Minor factual corrections, none fatal: (1) critic's 'config field/clamp' is wrong — no clamp line exists for the boolean; (2) file list missed en_us.json:8-9 (2 lang keys) and CLAUDE.md:135, which slightly widen the removal surface; (3) code LOC removed is ~20, not ~25; (4) the 8000-column inline freeze is a burst/backlog worst case (initial spiral scan, post-stall catch-up) — in steady state a toggled-off client drains small per-tick batches, so frame it as a reachable hazard, not typical-case harm; (5) it is a user-facing removal: anyone who pinned false (e.g. as a support workaround) silently reverts to off-thread on update — needs a changelog line. Honest counterweight to 'high' worth: the default-true path means keeping the knob costs almost nothing day-to-day, and some maintainers deliberately keep such support escape hatches — the proposed -D system property answers that at zero user-facing cost, but the call is discretionary.
