## Fabric + common Minecraft-API drift audit: 26.1.2 → 1.21.11 backport

### Method & authoritative sources
- Compared HEAD (`support/mc1.21.11` == v0.5.0, MC 26.1.2) against tag `v0.2.3` (last 1.21.11 release) file-by-file.
- **Two in-repo migration commits are authoritative for the actual rename set in this timeline**: `93615bf` ("Update Fabric mod to Minecraft 26.1", the full 1.21.11→26.1 Fabric migration) and `59e06a5` ("Bump Paper module to Minecraft 26.1.2", the NMS-side signature changes). Everything the forward migration touched must be inverted; everything it didn't touch and that existed at v0.2.3 is proven-compatible with 1.21.11.
- Key context: 26.1 is "the first fully unobfuscated release" — its runtime names ≈ 1.21.11 Mojang official mappings. That is why drift is tiny. **1.21.11 Mojang mappings in this timeline already use `Identifier` (not `ResourceLocation`)** — verified: `v0.2.3:fabric/.../payloads/*.java:7 import net.minecraft.resources.Identifier`, `Identifier.parse(...)` in v0.2.3 payload codecs and `LSSConfigMenu`.

## (b) Versioning model for the dual line

1. **Tag scheme:** `v<mod_version>+mc1.21.11` (e.g. `v0.5.0+mc1.21.11`) on `support/mc1.21.11`. `+` is legal in git refnames and matches the `v*` trigger. Never reuse a bare `vX.Y.Z` (already taken by main's line).

4. Anti-pattern to avoid: `v0.5.0-mc1.21.11` (hyphen = semver prerelease, sorts BEFORE 0.5.0 and breaks `--sort=-v:refname` ordering). Use `+` metadata.
5. CLAUDE.md's release pre-flight command must be branch-adjusted to pass the stripped `-Pmod_version` (the tag carries the suffix; the Gradle property must not).
